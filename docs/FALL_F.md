# fall -f — icantcode.today (Game Design Doc)

> **상태**: MVP 결정안 (2026-05-08).
> **범위**: `/game/fall-f` 단일 게임의 컨셉·메커니즘·UX 결정 사항 정리.
> **짝 문서**: 구현 플랜은 [`./FALL_F_PLAN.md`](./FALL_F_PLAN.md).
> **본 문서 사용법**:
> - **상단 §A / §B 만 보면** 의사결정 결과를 빠르게 파악할 수 있다.
> - **§1 ~ §5 는 상세 자료** — 결정 근거, 비교 옵션, ASCII 시뮬레이션,
>   대안 변주, 폐기된 안 등을 모두 보존. 구현 단계에서 디테일 참조용.

## Context — 왜 이걸 만드는가

icantcode.today 는 본질적으로 "API 가 죽었을 때만 살아나는" 서비스라
대기 시간이 단조롭다는 한계가 있다. API 가 다시 살아나기를 기다리는
사용자에게 **시간을 보낼 수 있는 가벼운 인터랙션**을 제공하면서도,
서비스의 핵심 미학("멀리서 보면 코딩 중인 것처럼 보여야 한다",
[`DESIGN_SYSTEM.md`](./DESIGN_SYSTEM.md) §1)을 훼손하지 않을 수 있는
방법으로 **터미널 풍 텍스트 게임**을 검토한다.

핵심 제약:

- 게임 화면이 보여도 **여전히 "터미널에서 빌드 로그가 흐르는" 인상**을
  유지해야 한다. (cf. [`SERVICE_PLAN.md`](./SERVICE_PLAN.md) §3.2)
- MulmaruMono 단일 폰트 + DejaVu Sans Mono 블록 글리프 서브셋(§2.5),
  박스 드로잉 문자, prompt 기호(`>`, `$`, `#`), 의미 있는 모션만 허용
  ([`DESIGN_SYSTEM.md`](./DESIGN_SYSTEM.md) §10.2 금지 목록 준수).
- 닉네임/UUID 세션 정책을 깨지 않는다. 게임에서 닉네임 강제 입력 금지.
- `prefers-reduced-motion` 존중.

## A. MVP 결정안 (Phase 1 — 첫 출시)

### A.1 게임 정체성

| 항목 | 결정 |
|---|---|
| 슬러그 | `/game/fall-f` |
| 표시 이름 | `fall -f` |
| 부제목 | `follow the fall` (영어 고정) |
| 장르 | Falldown (NS-Shaft 류) — 라인이 위로 흐르고 캐릭터는 떨어지며 살아남기 |

### A.2 라우팅 & 노출

- 기존 `/` (피드/랜딩) 그대로 유지.
- 신규: `/game` (카탈로그) + `/game/fall-f` (게임 인스턴스).
- 게임은 **API 상태와 완전 독립**. normal/down 모두에서 항상 진입 가능,
  게임 내에서 API 변화에 어떤 반응도 안 함.
- 헤더 nav 에 `[ play ]` 추가 + 세퍼레이터 `│` + 기존 `[ EN ] [ light ]`.
- 홈 회귀 = 좌측 타이틀(Logo + `username@icantcode.today`) 클릭.

### A.3 게임 메커니즘 (Pure Falldown)

- **입력**: ←/→ 만. 점프·호핑 없음.
- **물리**: 자유낙하 + 라인 위 착지 시 라인이 위로 흐르는 동안 함께 끌려감.
- **사망**: `[TIMEOUT]` (위로 밀려 사라짐) / `[SEGFAULT]` (모든 발판 통과 추락).
- **발판 정의**: 라인 전체 = 한 발판 (라인 끝 마커 ▌ 없음).
- **공백 = 빈 공간** (단어 사이 한 칸 포함). 모든 공백 위에서 떨어짐.
- **라인 그룹**: 라인 풀 데이터에 그룹 단위 정의 (`{ lines: [...], gap: 1|2 }`).
  그룹 X 범위 = 그룹 내 모든 라인의 발판 조각 합집합. 캐릭터는 그룹 가장
  윗줄 한 줄 위에 착지.
- **솔버블 보장 (R-1 + R-2)**: 짧은 라인 K 줄 연속 제한 + 인접 발판 X 인접성.
- **점수**: 라인 수.
- **엔딩**: G-1 무한 생존. 사망 시 결과 화면 (§A.6).
- **난이도 곡선**: 0–20s 1.0 line/sec → 60s+ 1.6 line/sec 캡 (선형/ease-in).

### A.4 캐릭터 & 입력 피드백

| 상태 | 글리프 |
|---|---|
| 정지 / 좌우 동시 | `█` (U+2588) |
| ← holding | `▌` (U+258C) |
| → holding | `▐` (U+2590) |

- 모두 모노스페이스 한 셀. X 좌표·충돌 판정에 영향 없음.
- 사망 시 `*` 또는 `x` 1프레임 후 사라짐.

### A.5 시작 흐름

- **Initial 화면**: `$ fall -f` prompt + 헤더 + 부제목 + KEYS/DEATH/best
  + `$ press [Enter] to start_` (한국어: `[Enter] 키를 눌러 시작_`).
- **Enter / Tap → 즉시 게임 시작.** 카운트다운 없음 (큰 글자 사이즈
  변화가 터미널 미학을 깨므로 폐기).
- **첫 라인 등장 흐름 (자연스러운 grace period)**:
  - t=0: 빈 화면 + 캐릭터 자유낙하 시작.
  - t≈0.5s: 첫 라인이 화면 아래에서 등장.
  - t≈1s+: 본 게임 페이스 진입.

### A.6 사망 결과 화면

`core dumped` 풍 가짜 스택트레이스:

```
[SEGFAULT] segmentation fault (core dumped)

  at line 142, score 99
  best  256
  in    fall_f::on_tick (game.ts:42)

[ > retry ]  [ > home ]
```

- 액션 = 재시작 / 홈 두 개. 디폴트 포커스 `[ > retry ]` (Enter 한 방).
- 점수 공유·피드 포스트 기능 없음 (게임 ↔ API 분리 정책).

### A.7 카탈로그 (`/game`)

```
$ ls -la /game/

-rwxr-xr-x  fall-f             2026-05  follow the fall
-rwxr-xr-x  (more coming...)

> _
```

### A.8 플랫폼 / 반응형 / 일시정지

- **데스크탑**: ←/→ 키.
- **모바일**: 화면 좌/우 절반 영역 탭/홀드 = ←/→. 두 영역 동시 터치 무시.
- **반응형**: 한 줄 칸 수는 **현재 브라우저 폭** 기준으로 신규 라인에만 적용.
  기존 라인은 등장 시점의 폭으로 고정 (창을 줄이면 기존 라인이 화면 밖으로
  잘려 죽을 수 있음 — 의도적 감안).
- **가로 스크롤 절대 발생 안 함.** 게임 컨테이너에 `overflow-x: hidden`
  (또는 `overflow: clip`) 적용. 큰 창에서 시작해 도중에 창을 줄여도
  스크롤바 대신 화면 밖 영역은 시각적으로 잘림. 세로 스크롤도 게임
  영역에서는 발생 안 함.
- **캐릭터 X 좌표는 현재 화면 폭 안으로 제한.** 캐릭터는 화면 밖으로
  나갈 수 없음. 따라서 폭 너머로 잘린 발판 영역은 결과적으로 도달 불가능
  — 사용자가 본 발판만 게임 룰에 유효.
- **한 화면 라인 수**: 화면 높이 / line-height (~24px) 자동 (대략 25~30줄).
- **탭 hidden 자동 일시정지**: `visibilitychange` 로 게임 루프 정지,
  복귀 시 같은 상태 이어서 재개. Esc 키는 무시 (별도 일시정지 모달 없음).

### A.9 동적 발판 (압박 그룹)

- 채택 변주 2종:
  - **점 사라짐**: `loading........` 우측에서 한 글자씩 짧아짐.
  - **프로그레스 바 비우기**: `[████████]` → `[███████░]` → ... 우측 빈칸화.
- 캐릭터가 발판 X 범위 밖으로 밀려나면 **자동 slide** (벽 충돌 한도 안에서).
- 라인 풀에서 압박 그룹 비율 10~20%.

### A.10 시각·미디어 정책

- 라인 끝 마커 ▌ **없음**. 텍스트 마지막 글자가 발판 우측 끝.
- 라인 그룹 사이 빈 줄 1~2칸 (gap, 라인 풀에서 결정).
- 사운드 **없음**.
- 모션: 라인 등장은 한 번에 (타이핑 효과 없음). 캐릭터 정지 시 커서 깜빡임.
- `prefers-reduced-motion`: 라인 스크롤 step 단위 즉시 이동.
- 다크/라이트 모드 모두 CSS 변수 (`--primary`, `--foreground`, `--destructive`
  등) 그대로 활용. 별도 분기 없음.

### A.11 폰트

- **MulmaruMono** (한글/영문) + **DejaVu Sans Mono 서브셋** (Block Elements
  + Box Drawing, U+2500–U+259F).
- CSS `unicode-range` 로 자동 분기.
- 서브셋 woff2 5–15KB. `font-display: swap`. **preload 는 안 함** (운영 중
  LCP 측정 후 필요시 도입).

### A.12 i18n 정책

| 종류 | 정책 |
|---|---|
| 시스템 토큰 (`[TIMEOUT]`, `[SEGFAULT]`, `core dumped`) | 영어 고정 |
| 명령어·prompt (`$ fall -f`, `$ ls -la`) | 영어 고정 |
| 라인 풀 빌드 로그 | 영어 고정 |
| 부제목 (`follow the fall`) | 영어 고정 |
| 게임 설명·라벨 (`KEYS`, `DEATH`, `best`, `←/→ move`, `pushed off the top`) | **ko/en 분기** |
| 액션 (`[ > retry ]`, `press [Enter] to start`) | **ko/en 분기** |

기준: "터미널이 출력하는 것 = 영어, 사람이 사람에게 안내하는 것 = 분기."

### A.13 세션·점수 영속성

- 닉네임/UUID 정책 변경 없음 ([`SERVICE_PLAN.md`](./SERVICE_PLAN.md) §4.3 그대로).
- best score = **세션 메모리만**. 페이지 새로고침 시 초기화 (현재 서비스의
  무로그인 정책 일치).

## 1. 사용자 원안 (Initial concept)

> 횡스크롤 플랫폼 게임을 수직 스크롤로. 터미널에 한 줄씩 글이 올라오는
> 것처럼 길이가 다른 텍스트가 나오고, 텍스트의 오른쪽 끝나는 지점이
> 플랫폼이 되는 느낌. 멀리서 보면 그냥 터미널에서 작업 중인 것처럼
> 보였으면 한다. 진행 방식·룰·골은 아직 미정.

### 시각 컨셉 스케치

라인 끝에 마커는 없다 (§2.17). **텍스트의 마지막 글자가 그대로 발판의
우측 끝**. 라인이 짧으면 발판도 왼쪽에, 라인이 길면 오른쪽까지 이어진다.
캐릭터는 `█` (§2.5). **발판 라인 사이에는 한 줄씩 빈 줄을 끼워넣어
캐릭터가 설 수 있는 자리를 만든다 (§1.5)**.

```
┌──────────────────────────────────────────────┐
│ $ npm run build                              │  ← 발판
│                                              │  ← 빈 줄 (█ 자리)
│ ▸ vite v6.0.7 building for production...     │  ← 발판
│                                              │
│ ✓ 142 modules transformed.                   │
│                                              │
│ ▸ rendering chunks...        █               │  ← █ 가 다음 발판 한 줄 위
│ ▸ computing gzip size...                     │  ← 발판 (█ 의 발 아래)
│                                              │
│ ✓ built in 8.42s                             │
│                                              │
│ $ tsc --noEmit                               │
│                                              │
│ src/foo.ts(12,3): error TS2322:              │
│                                              │
│ $ _                                          │
└──────────────────────────────────────────────┘
```

> 멀리서 보면 정말 그냥 빌드 로그가 흐르는 터미널 (라인 사이 약간 듬성
> 한 정도). 가까이서 보면 `█` 가 라인 한 줄 위 빈 칸에 머물러 있다.

> ⚠ **§3.3 발판 정의의 미해결 쟁점**: 사용자 원안의 "텍스트 오른쪽
> 끝나는 **지점**" 이라는 표현은 *점/좁은 영역* 의 어감이 강하다. 따라서
> §3.3 의 두 해석 중 어느 쪽이 의도에 더 가까운지는 §3.3 의 표를 보고
> 다시 합의 필요. 본 §1 스케치는 둘 중 어느 해석이어도 (라인 전체이든
> 끝 일부이든) "라인이 짧으면 발판도 왼쪽" 이라는 핵심 시각 규칙은
> 동일하다.

### 1.1 실존 유사 게임 (장르 레퍼런스)

본 게임의 메커니즘은 다음 실존 게임의 텍스트 버전이라고 보면 된다.
플레이 영상은 키워드로 직접 검색.

- **NS-Shaft** (1996, PC) — 가장 직접적인 조상. 가로 발판들이 위로
  흐르고 플레이어가 발판 사이를 빠져나가며 아래로 내려간다. 검색 키워드:
  `NS-Shaft gameplay`.
- **Fall Down (Falldown)** (2000년대 초, 플래시 게임) — 같은 장르.
  검색 키워드: `Falldown flash game`.
- 그 외 모바일 후속작들 — 같은 메커니즘이 여러 번 리메이크됨.

`@` 가 공/캐릭터에 해당하고, 가로 발판이 텍스트 라인에 해당한다고
보면 된다.

### 1.2 게임 진행 단계별 컷

핵심 규칙: **"라인은 위로 흐른다. 가만히 있으면 화면 위로 밀려 죽으니,
←/→ 로 발판 끝을 벗어나 아래 새 라인으로 떨어지며 살아남는다."**

#### 모노스페이스 그리드 표현 규칙 (먼저 읽어주세요)

ASCII 그림에서 "█ 가 발판 위에 서 있다" 의 정확한 정의:

- 발판은 **텍스트가 그려진 줄 자체** 이며, 그 줄에는 다른 문자가 들어
  가지 않는다. 발판 X 범위 = 텍스트 시작 X ~ 마지막 글자 X.
- █ 가 발판 위에 서 있다 = █ 가 발판 줄의 **정확히 한 줄 위**에 있고,
  █ 의 X 좌표가 발판 X 범위 안에 있을 때.
- █ 가 발판 줄과 **같은 줄**에 있으면 그건 옆에 떠 있는 상태 (중력으로
  다음 프레임에 떨어짐) 이며, 착지가 아니다.
- 떨어질 때는 발판 줄에서 한 줄 더 아래로 위치가 바뀌고, 그 줄에서 또
  발판이 있는지 X 매칭으로 판정한다.

```
✗ 같은 줄에 붙으면 착지 아님:        ✓ 한 줄 위 + X 범위 안 = 착지:
│ ✓ 142 modules█                   │     │             █                │
│                                  │     │ ✓ 142 modules                │
```

#### T=0 ~ T=4 진행

```

│                                        │
│                                        │
│                                        │
│                                        │

│                                        │
│                                        │

        다음 프레임 떨어짐.
```

화면 위로 밀려 죽고(TIMEOUT), 떨어질 때 다음 발판 X 범위에 못 맞추면
계속 통과해 바닥 추락(SEGFAULT)"** 두 위협이 동시에 가해져 긴장감을
만든다.

### 1.3 라인 생성과 솔버블 보장

원안 그대로 "라인이 랜덤 길이로 등장" 하면 두 가지 게임 디자인 이슈가
생긴다.

#### 1.3a 좌측에도 빈 공간이 있어야 한다

라인이 모두 X=0 부터 시작하면 **왼쪽 낙하가 영영 불가능** 해 게임이
오른쪽 일변도가 된다. 해결: 실제 터미널 출력의 자연스러운 **들여쓰기
변주** 를 활용한다.

```
$ npm run build                 ← X=0~14 (들여쓰기 없음)
  ▸ vite v6.0.7                 ← X=2~14 (들여쓰기 2)
  ✓ 142 modules                 ← X=2~14 (들여쓰기 2)
    transforming (12)...        ← X=4~22 (들여쓰기 4)
$ tsc --noEmit                  ← X=0~13
src/foo.ts(12,3): error...      ← X=0~24
```

이렇게 하면 발판마다 (시작 X, 끝 X) 두 값이 모두 다양해져 **좌우
양쪽으로 떨어지는 게임플레이가 가능**하다. 자연스러운 빌드 로그 인상도
유지된다.

#### 1.3b 솔버블 보장 (Solvability) — Pure Falldown 의 핵심 제약

입력이 ←/→ 만이고 점프가 없으므로, 다음 발판이 내 X 범위 밖이면 그
대로 죽는다. 따라서 라인 생성 알고리즘은 다음을 보장해야 한다.

- **R-1. 연속 짧은 라인 제한**: 짧은 라인(폭의 30% 이하)이 K 줄 연속
  나오면 다음 줄은 강제로 긴 라인.
- **R-2. 인접 발판 X 인접성**: 새 라인 등장 시, 직전 N 줄의 발판 영역
  중 적어도 하나는 새 라인의 발판 영역과 **X 축에서 일정 거리(예: 5칸)
  이내** 로 겹치거나 인접. "어디 위에 있든 다음 발판으로 갈 수 있는"
  보장.
- **R-3. 플레이어 X 인지(옵션)**: 새 라인 X 범위가 현재 플레이어의
  X ± N 안에 항상 한 줄은 등장하도록 가중치 조정.

##### 보장 강도 옵션

| 강도 | 룰 | 게임플레이 영향 |
|---|---|---|
| 약함 | R-1 만 | 랜덤성 강함, 가끔 운으로 죽음 |
| 중간 | R-1 + R-2 | 랜덤성 유지하면서 솔버블, **권장** |
| 강함 | R-1 + R-2 + R-3 | 거의 죽지 않음. 너무 쉬움 가능 |

> 잠정 채택: **중간 강도 (R-1 + R-2).** 라인 풀에서 무작위로 뽑되,
> 직전 줄과의 X 인접 제약을 만족할 때까지 재추첨하는 단순 알고리즘.

#### 1.3c 메커니즘 변경 옵션 (백업 카드)

위 보장이 어색하거나 구현이 복잡하면, **카드 #2 Hop Falldown** (점프
없이 작은 자동 호핑) 또는 **카드 #3 Manual Jump** 으로 메커니즘을
바꾸는 것도 방법이다. 점프가 있으면 발판 간 X 갭이 어느 정도 있어도
메꿀 수 있어 솔버블 부담이 크게 준다. Pure Falldown 의 위장 효과는
일부 양보하는 트레이드오프.

### 1.4 발판은 글자 단위 — 공백은 빈 공간

지금까지의 가정 ("라인 전체 = 한 발판") 을 한 단계 더 미세화한다.
한 줄 안에서도 **글자가 있는 곳만 발판, 공백은 모두 빈 공간** 으로
처리하면 "텍스트 구조 자체가 게임 발판 패턴이 된다" 는 독창적 디자인이
된다.

#### 채택 룰 — 모든 공백 = 빈 공간 (룰 B)

- **공백 1칸이든 2칸 이상이든 모두 빈 공간** (캐릭터 통과 가능).
- 게임 룰: "공백 위에서는 떨어진다" 한 줄로 끝나는 일관된 직관.
- 자연스러운 결과:
  - `$ npm run build` → 발판 4조각 (`$`, `npm`, `run`, `build`).
    단어 사이 한 칸 공백도 통로.
  - `drwxr-xr-x  15 root root  4096  node_modules` → 발판 6조각.
    컬럼 사이 두 칸 공백은 명확한 통로.
  - 들여쓰기 (4칸 이상) → 좌측에 자연스러운 빈 공간.

```
드물게 나오는 통로 라인 예 (ls -la 출력):

drwxr-xr-x  15 root root  4096  node_modules
^^^^^^^^^^  ^^ ^^^^ ^^^^  ^^^^  ^^^^^^^^^^^^
   발판     발  발판  발판  발판     발판
          ↕           ↕     ↕     ← 컬럼 사이 두 칸 공백 = 통로

drwxr-xr-x   8 admin app   2048  dist-prod
^^^^^^^^^^   ^ ^^^^^ ^^^   ^^^^  ^^^^^^^^^
            ↕                   ← 위 줄과 통로 X 가 겹치면 연속 낙하
```

#### 룰 변경 이력 (룰 A 폐기)

이전 논의에서 "한 칸 공백은 발판, 두 칸 이상만 빈 공간" (룰 A) 을 잠정
채택했으나, 다음 두 약점으로 폐기했다.

- **시각 구분 어려움**: 모노스페이스에서 한 칸/두 칸 공백을 한눈에
  구분하려면 매번 헤아려야 한다 — 학습 비용·인지 부담 큼.
- **"비어있는데 안 떨어지는" 어색함**: 한 칸 공백 위에 캐릭터가 서 있는
  상태가 직관에 어긋난다.

룰 B 는 "공백 = 떨어진다" 한 줄로 끝나는 일관된 룰이라 학습 비용이
사실상 0.

#### 룰 B 의 알려진 트레이드오프와 보완책

- **단어 사이 한 칸 통로가 좁다 → 정밀 조작 어렵다.**
  - 캐릭터는 모노스페이스 한 칸 너비라 X 위치가 명확히 보인다. 의지가
    있으면 한 칸 통로도 노릴 수 있다.
  - 의도치 않게 한 칸 통로로 떨어져도 **죽음으로 직결되지 않는다** —
    솔버블 보장 (§1.3b) 이 다음 발판 도달성을 보장.
- **라인 풀 디자인 부담** — 단어 사이 통로가 과도하지 않도록 긴 단어
  위주 라인 (`▸ vite-plugin-react-swc`) 도 섞고, 컬럼 정렬 라인
  (`ls -la`, `ps aux`) 은 의도적 통로 변주로 활용.
- **솔버블 보장 알고리즘 정교화 필요** — 발판 조각 수가 라인당 1~10개로
  늘어남. R-1 + R-2 알고리즘은 동일 적용, 단 인접 발판 비교 시 모든
  조각을 검사.

### 1.5 라인 그룹 + 빈 줄 변주 정책

#### 문제

캐릭터 `█` 가 발판 위에 서려면 발판 줄의 **한 줄 위에 빈 자리** 가 필요
하다. 발판 라인이 화면 줄 단위로 무조건 연속되면 그 사이엔 캐릭터가
설 자리가 없어 죽은 발판이 되고, 반대로 항상 1:1 로 "발판 한 줄 다음
빈 줄 한 줄" 만 반복하면 화면이 단조롭다. 또 `ls -la` 같은 컬럼 정렬
출력은 라인이 연속으로 나오는 게 자연스럽다.

#### 채택 — 그룹 기반 변주

- **그룹 (group)** = 빈 줄 없이 연속된 발판 라인의 묶음. 라인 풀
  데이터가 그룹 단위로 정의되어 있다.
- **그룹 X 범위** = 그룹 내 모든 라인의 발판 조각들의 X 좌표 합집합.
  같은 X 컬럼에 그룹 안 어느 한 줄이라도 글자가 있으면 그 X 는 발판.
- **캐릭터 착지** = 그룹의 가장 윗줄 한 줄 위 (빈 줄) 에서. 그룹 X
  범위 안이면 머물고, 밖으로 가면 떨어짐. 떨어질 때는 그룹의 가장
  아랫줄 다음 빈 줄로 통과 후 자유낙하.
- **그룹 사이 빈 줄 수 (`gap`)** = 라인 풀 데이터로 1~2 줄 사이에서
  변주. (0 이면 그룹이 합쳐져 의미 없음.)

#### 그룹 예시

```
{ kind: "single",  lines: ["$ npm run build"], gap: 1 }
{ kind: "ls-la",   lines: ["total 8",
                           "drwxr-xr-x  15 root root  4096  node_modules",
                           "drwxr-xr-x   8 admin app  2048  dist-prod"], gap: 2 }
{ kind: "vite",    lines: ["▸ vite v6.0.7 building...",
                           "✓ 142 modules transformed.",
                           "▸ rendering chunks...",
                           "✓ built in 8.42s"], gap: 1 }
{ kind: "tsc-err", lines: ["src/foo.ts(12,3): error TS2322:",
                           "  Type 'string' is not assignable to type 'number'."], gap: 1 }
```

화면에 등장하면:

```
$ npm run build               ← group 1, 1줄짜리
                              ← gap=1
total 8                       ← group 2 시작
drwxr-xr-x  15 root root  4096  node_modules
drwxr-xr-x   8 admin app  2048  dist-prod
                              ← gap=2
                              ← gap=2 (두 번째 줄)
$ vite build                  ← group 3 시작
▸ vite v6.0.7 building...
✓ 142 modules transformed.
▸ rendering chunks...
✓ built in 8.42s
                              ← gap=1
src/foo.ts(12,3): error TS2322:   ← group 4
  Type 'string' is not assignable to type 'number'.
```

#### 게임 효과

- **두꺼운 그룹** (ls-la, vite 빌드 진행) — 좌우 폭이 넓어 안전한 휴식
  구간. 잠시 숨 고르고 다음 그룹을 준비.
- **1줄 그룹** — 좁은 발판, 빠른 결정 필요. 게임 페이스를 끌어올림.
- **gap 변주 (1~2줄)** — 두 줄 갭에서는 자유낙하 시간이 길어져 X 위치를
  더 정밀하게 조정해야 함. 한 줄 갭은 빠른 전환.

#### 옵션 B (미래 최적화) — line-height absolute positioning

- 실제 게임 렌더링 시 CSS `line-height` 를 1.6~2.0 으로 잡고 캐릭터
  글리프를 absolute position 으로 라인 Y 좌표 위 일부 픽셀 위치에 그림.
- 위 그룹 + 빈 줄 룰은 그대로 유지. ASCII 문서상 빈 줄이 실제 렌더에선
  픽셀 간격으로 표현됨.
- 위장 효과 강화. 구현 부담 약간 ↑. 출시 후 검토.

#### 본 문서 ASCII 그림의 일관성

본 문서 §1, §1.2 의 단계별 컷은 단순화를 위해 "1줄 그룹 + gap=1"
케이스만 그렸다. 실제 게임에선 위처럼 그룹 크기와 gap 이 라인 풀에서
랜덤으로 뽑혀 변주된다. §1.4 의 ls-la 분석 그림이 "여러 줄 그룹 +
gap=2" 의 자연스러운 사례.

### 1.6 동적 발판 (시간에 따라 변하는 라인)

지금까지 가정한 라인은 모두 **정적**이었다. 한 번 등장하면 텍스트가
고정되고 위로 흐르기만 했다. 여기에 **동적 발판** — 시간에 따라 텍스트
길이가 변해 발판 X 범위 자체가 변하는 라인 — 을 추가해 시간 압박을
더하는 변주를 도입한다.

#### 디자인 후보 (모두 터미널 스타일)

| 패턴 | 예시 | 변화 |
|---|---|---|
| **점 사라짐** | `loading........` → `loading.......` → ... → `loading` | 우측에서 한 글자씩 짧아짐 |
| **타이핑 거꾸로** | `installing deps...` → `installing dep` → … | 우측 끝부터 한 글자씩 삭제 |
| **프로그레스 바 비우기** | `[████████]` → `[███████░]` → `[██████░░]` | 우측이 한 칸씩 빈칸으로 |
| **프로그레스 바 채우기** | `[░░░░░░░░]` → `[█░░░░░░░]` → `[██░░░░░░]` | 좌측에서 자라남 (성장) |
| **카운트다운** | `[wait 5s]` → `[wait 4s]` → … → 사라짐 | 글자 수 유지, 0 도달 시 그룹 소멸 |
| **사라지는 에러** | `error: file not found` → `rror: file not found` → … | 좌측에서 한 글자씩 사라짐 |
| **맥동** | `[==>      ]` → `[====>    ]` → `[==>      ]` 반복 | 늘어났다 줄어들기 사이클 |

#### 발판 끝으로 밀려난 캐릭터 처리 — 자동 slide (잠정 채택)

발판이 줄어들 때 캐릭터가 발판 X 범위 밖으로 밀려나는 상황의 두 옵션:

| 옵션 | 동작 | 평가 |
|---|---|---|
| **자동 slide (권장)** | 발판이 줄어들 때 캐릭터를 자동으로 발판 안쪽 한 칸 안으로 밀어줌 | "줄어드는 발판이 캐릭터를 밀어내는" 자연스러운 시각 효과 |
| **그냥 떨어짐** | 캐릭터 X 가 발판 밖이 되면 그대로 추락 | 도전적이지만 학습 곡선↑ |

자동 slide 채택. 단 캐릭터 좌우에 다른 발판이 있다면 그 X 위치까지만
slide (벽 충돌 처리). 슬라이드도 한계가 있어 "결국 발판이 사라지면 떨어
진다" 는 룰은 동일.

#### 라인 풀 데이터 구조 확장

기존 정적 그룹과 호환되도록 동적 그룹은 메타데이터로 구분:

```
정적 그룹:
{ kind: "single", lines: ["$ npm run build"], gap: 1 }

동적 그룹 (점 사라짐):
{ kind: "loading", dynamic: "shrink-right",
  base: "loading", suffix_init: "........",
  shrink_per_sec: 2, gap: 1 }

동적 그룹 (프로그레스 바 비우기):
{ kind: "progress", dynamic: "fill-right",
  template: "[XXXXXXXX]", filled: "█", empty: "░",
  duration_sec: 4, gap: 1 }

동적 그룹 (카운트다운):
{ kind: "countdown", dynamic: "countdown",
  template: "[wait Ns]", from: 5, to: 0,
  step_per_sec: 1, expire_action: "destroy", gap: 1 }
```

#### 게임 디자인 측면 (압박 그룹 비율)

- 동적 발판은 **압박 그룹** 카테고리. 라인 풀에서 등장 비율 10~20%.
- 너무 자주 나오면 게임이 압박만 강해져 단조로움.
- 정적 그룹과 섞어 페이스 조절: 정적 안전 구간 → 동적 압박 → 정적 휴식
  → … 의 리듬.

#### 첫 출시 채택 — 점 사라짐 + 프로그레스 바 비우기

- **점 사라짐** (단순, 직관, 학습 비용 0)
- **프로그레스 바 비우기** (시각적 강력, 시간 흐름 명확)
- 나머지 (타이핑 거꾸로, 카운트다운, 사라지는 에러, 맥동) 는 §1.6a
  미래 변주 후보로 보존. 출시 후 게임플레이 검증 후 점진 추가.

#### 라인 풀 디자인 함의

룰 A 채택 시 라인 풀 구성에 다음 변주를 의도적으로 섞으면 게임 흐름이
풍부해진다.

- **컬럼 정렬 라인** — `ls -la`, `ps aux`, `top`, `ifconfig` 출력 풍.
  여러 줄에 걸쳐 통로 X 좌표가 비슷하게 정렬되면 "수직 자유낙하 구간" 을
  만들어 시원한 액션감.
- **들여쓰기 깊은 코드 라인** — `    transforming src/main.tsx` 처럼
  좌측 빈 공간이 큰 라인. 좌측 낙하 통로.
- **JSON / TOML 풍** — `  "key": "value",` 처럼 콜론·콤마 사이 변주가
  있는 라인.
- **단순 명령 라인** — `$ npm run build` 처럼 한 발판으로 통합되는
  평범한 라인. 휴식 구간.

라인 풀의 비율을 조절해 난이도와 페이스를 만든다.

#### §1.3b 솔버블 보장 갱신

룰 A 채택 시 솔버블 보장은 다음으로 확장된다:

- 발판은 한 줄 안에서도 여러 조각. 각 조각의 X 범위 = (시작 X, 끝 X).
- R-2 (인접 발판 X 인접성) 는 "직전 N 줄의 **모든 조각 합집합**" 과
  새 줄의 모든 조각 합집합 간의 인접성으로 일반화.
- 구현 부담은 크지 않음 (조각 수는 라인당 1~6개 수준).

> 만약 이 흐름이 직관과 다르다면 다음 대안 방향이 있다 — §3.2 옵션 B
> (Doodle Jump 식: 라인이 아래로 흐르고 @ 가 위로 점프), 또는 컨셉
> 자체를 다른 장르(타이핑, 어드벤처 등)로 변경.

## 2. 결정된 사항

### 2.1 라우팅

- **단일 라우트 정책 폐기.** 기존 `/` 는 그대로 피드/랜딩 유지.
- 게임은 별도 경로를 신설: **`/game`** (게임 카탈로그 / 리스트) 와
  **`/game/:slug`** (개별 게임 인스턴스).
- `/play` 가 아닌 `/game` 을 채택. 이유:
  - 카탈로그/리스트 페이지의 의미상 명사형이 자연스러움 (`/play` 는
    동사라 리스트와 의미 충돌, 단일 게임 직링크에는 어울리지만 리스트엔
    어색).
  - 터미널 비유로도 `$ ls /game/` → `dist-prod falldown.exe` 같은 구조가
    자연스럽다.
  - 짧고 한국어 사용자에게도 명확.
- 슬러그 예시: `/game/falldown`, `/game/snake`, `/game/typing-trial`.
- normal 상태에서도 접근 가능. "일하기 싫을 때 잠깐 노는 공간" 도 OK.
- 향후 게임이 늘어날 수 있는 구조 (`/game/` 카탈로그 리스트가 가장 위
  계층).

> 영향: [`SERVICE_PLAN.md`](./SERVICE_PLAN.md) §4.1 의 "단일 라우트"
> 문구를 "기본 페이지는 단일 라우트지만 게임은 별도 경로" 로 갱신해야
> 한다. [`ARCHITECTURE.md`](./ARCHITECTURE.md) 의 라우팅 섹션도 동일.

### 2.2 헤더 메뉴 디자인

- 홈으로 돌아가는 동선은 **페이지 타이틀(Logo + `username@icantcode.today`)
  클릭** 으로 충분. 별도의 "Home" 항목은 추가하지 않는다.
- 우측 nav 에 **`[ play ]`** 항목을 추가. 현재 우측 항목은 (lang 토글,
  theme 토글) 두 개 → (`play`, lang, theme) 세 개로 늘어남.
- 두 그룹 사이에 **세퍼레이터**(`│` U+2502, 또는 `·`)를 넣어 메뉴
  그룹과 환경설정 그룹을 시각적으로 구분.
- 라벨은 **`[ play ]`** (동사형) — 라우트가 `/game` 이어도 메뉴 라벨은
  "들어가서 한다" 를 강조하는 동사가 더 자연스럽다.

```
좌측                                 우측
guest@icantcode.today:~$_   [ play ] │ [ EN ] [ light ]
                            ─────────  ─────────────────
                            게임 메뉴   기존 환경설정
```

### 2.3 노출 정책

- 게임은 API 상태와 **완전히 독립**. 진입·플레이 모두 API 상태와 무관.
- 게임 내부에서 API 상태 변화에 어떤 반응도 하지 않는다 (토스트, 안내,
  자동 종료 모두 없음). 게임은 게임만 다룬다.

### 2.4 골·엔딩 구조 (G-1 무한 생존)

- **G-1 무한 생존** 채택. 죽을 때까지 끝없이 진행.
- 점수 = 살아남은 라인 수 또는 누적 시간.
- 사망 시 결과 화면(§2.13)에 점수 표시 + 재시작/홈 액션.
- 최고 기록은 세션 메모리에 보관 (서비스의 무로그인 정책 일치).
- API 상태 기반 자동 종료·하이브리드 모드는 채택하지 않는다 (§2.3).

### 2.5 플레이어 캐릭터 글리프

- 베이스 글리프: **`█`** (터미널 커서, U+2588 FULL BLOCK).
  "입력 커서가 로그 사이를 뛰어다닌다" 는 메타포가 가장 강력.
- 라인 끝 마커 `▌` 를 사용하지 않으므로(§2.17) 모양 충돌 우려 없음.

#### 입력 피드백 — 좌우 키 holding 시 절반 블록

키 입력 상태에 따라 캐릭터 글리프를 동적으로 교체해 입력 피드백을
캐릭터 자체에 통합한다. 모두 모노스페이스 한 셀 너비라 X 좌표·충돌
판정에는 영향 없음. 시각적으로만 셀의 절반/전체가 칠해진다.

| 상태 | 글리프 | 코드포인트 | 시각 |
|---|---|---|---|
| 입력 없음 / 좌우 동시 누름 | `█` | U+2588 FULL BLOCK | 셀 전체 |
| ← holding | `▌` | U+258C LEFT HALF BLOCK | 셀 좌측 절반 |
| → holding | `▐` | U+2590 RIGHT HALF BLOCK | 셀 우측 절반 |

```
정지/중립          ← holding         → holding

│      █      │   │      ▌      │   │      ▐      │
│ $ build     │   │ $ build     │   │ $ build     │
```

- 두 키 동시 holding → 입력 무시, `█` 로 폴백.
- 모바일 좌/우 탭 영역 입력에도 동일 매핑 (탭/홀드 영역에 따라 ▌/▐).

#### 추후 추가 변주 (옵션)

같은 블록 글리프 시리즈로 다른 상태도 확장 가능. 출시 후 컨트롤 감각이
약하면 도입.

| 상태 | 글리프 | 의미 |
|---|---|---|
| 자유낙하 | `▄` U+2584 (하반) | 아래로 무게 |
| 점프 중 (메커니즘 변경 시) | `▀` U+2580 (상반) | 위로 무게 |
| 사망 | `*` 또는 `x` | 1프레임 충돌 후 사라짐 |

#### 폰트 호환성 — 검사 결과와 정책

**검사 결과** (fontTools / cmap 분석, 2026-05-08):

```
Block Elements (U+2580–U+259F)  →  MulmaruMono 미지원 (전부 MISSING)
Box Drawing    (U+2500–U+257F)  →  MulmaruMono 미지원 (전부 MISSING)
```

즉 `█ ▌ ▐ ┌ │ └` 같은 글리프는 MulmaruMono 가 직접 그릴 수 없다.
현재는 fallback chain (Menlo/Consolas/Courier New) 이 받아주지만
**OS 마다 받아주는 폰트가 달라 메트릭(폭·높이) 차이가 시스템마다 달라
지는 문제** 가 있다 (특히 Linux 사용자).

**채택 정책 — DejaVu Sans Mono 서브셋 + `unicode-range`**:

OS 무관 일관성을 위해 Block Elements + Box Drawing 두 블록만 추출한
DejaVu Sans Mono 서브셋을 self-host. CSS `@font-face` 의 `unicode-range`
디스크립터로 해당 글리프만 자동으로 이 폰트가 받음. 한글/영문은 그대로
MulmaruMono.

```css
@font-face {
  font-family: 'BlockGlyphs';
  src: url('/fonts/dejavu-blocks.woff2') format('woff2');
  unicode-range: U+2500-259F;
  font-display: swap;
}
body {
  font-family: 'MulmaruMono', 'BlockGlyphs', 'MulmaruMono Fallback', monospace;
}
```

- **용량**: 160 글리프 서브셋, woff2 압축 후 5–15KB 예상.
- **라이선스**: DejaVu fonts License (Bitstream Vera 기반, BSD-style).
  자유 사용·수정·임베드 OK. `public/fonts/LICENSE_DejaVu.txt` 동봉.
- **로딩**: `<link rel="preload">` + `font-display: swap`. LCP 영향 미미.
- **영향 범위**: 본 게임뿐 아니라 현재 서비스의 박스 드로잉 카드, 모든
  `┌─┐│└─┘` 표시도 OS 무관 일관 렌더링 됨. 부수 효과로 서비스 전반의
  시각 일관성↑.

**작업 단계** (구현 시):

1. 원본 DejaVu Sans Mono TTF 다운로드 (DejaVu 공식).
2. `pyftsubset --unicodes=U+2500-25FF --flavor=woff2` 로 서브셋.
3. `public/fonts/dejavu-blocks.woff2` + `public/fonts/LICENSE_DejaVu.txt`
   배치.
4. `src/styles/globals.css` (또는 적절한 위치) 에 `@font-face` +
   `unicode-range` 추가.
5. font-family chain 갱신 (DESIGN_SYSTEM §3.1 도 같이 갱신).

**`<link rel="preload">` 는 추가하지 않는다 (잠정).** 박스 드로잉이
메인에서도 쓰이지만 현재 fallback 이 받아주고 있어 시각적으로 정상
동작 중이며, preload 는 게임을 하지 않는 사용자에게도 비용을 부담시킨다.
`@font-face` + `font-display: swap` 만으로 첫 렌더는 fallback, 다운로드
완료 후 BlockGlyphs 로 swap 되는 흐름이 자원 효율 측면에서 적절. 운영
중 LCP/CLS 측정 결과 preload 가 정말 필요하면 그때 추가.

### 2.18 시작 화면 레이아웃

§2.9 의 시작 트리거 (Enter → 즉시 시작) 에 대한 시각 시안.
카운트다운은 채택하지 않으므로 화면은 Initial → 게임 두 단계뿐.

#### Initial 화면 (진입 직후)

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  $ fall -f                                               │
│                                                          │
│  ── fall -f v1.0.0 — follow the fall ─────────────────   │
│                                                          │
│  KEYS:   ←/→   move                                      │
│  DEATH:  [TIMEOUT]   pushed off the top                  │
│          [SEGFAULT]  fell through all platforms          │
│                                                          │
│  best:   256 lines                                       │
│                                                          │
│  $ press [Enter] to start_                               │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

- 좌측 정렬 + 모노스페이스 + 박스 프레임 = 터미널 출력 한 화면.
- 첫 줄 `$ fall -f` 는 명령어 실행 prompt. 그 다음 헤더가 그 명령어의
  배너 출력처럼 이어짐.
- `── fall -f v1.0.0 — follow the fall ──` 헤더 라인은 git/npm 출력
  스타일 + 부제목 한 줄로 결합.
- KEYS / DEATH 블록은 man page 인상. 모두 영어 토큰.
- `best:` 줄은 세션 메모리에 기록이 있을 때만 노출 (첫 진입 시 생략).
- 마지막 줄의 `_` 는 깜빡이는 커서 (DESIGN_SYSTEM §4.3).
- 모바일: `press [Enter]` → `tap to start`.
- i18n: §2.19 정책 적용. 시스템 토큰·명령어 풍 텍스트는 영어 고정.
  사람이 읽는 안내 문구가 추가될 경우만 ko/en 분기.

#### Enter / Tap 직후 — 즉시 게임 시작

Initial 화면이 사라지면서 게임 화면이 곧장 시작된다. 같은 프레임에서
첫 라인이 화면 아래에서 등장하고 캐릭터(`█`)가 화면 상단에서 자유낙하를
시작. 큰 글자 카운트다운 등 별도 인터스티셜 화면은 없다.

전환은 fade 또는 즉시 cut 둘 다 허용 (구현 단계에서 결정).
`prefers-reduced-motion` 시 즉시 cut.

### 2.19 UI 카피 다국어 분기 정책

게임 화면은 "터미널 출력 풍" 인 부분과 "사용자가 읽는 안내" 인 부분이
섞여 있다. 둘을 다음 기준으로 구분한다.

| 텍스트 종류 | 정책 | 예시 |
|---|---|---|
| 시스템 토큰·exit 코드 | **영어 고정** | `[TIMEOUT]`, `[SEGFAULT]`, `core dumped`, `(core dumped)`, `fall_f::on_tick` |
| 명령어 라인·prompt | **영어 고정** | `$ fall -f`, `$ ls -la /game/`, `> _` |
| 라인 풀 빌드 로그 | **영어 고정** | `$ npm run build`, `▸ vite v6.0.7 building...` 등 모든 라인 |
| 부제목 (§2.8) | **영어 고정** | `follow the fall` |
| 게임 설명·안내·라벨 | **ko/en 분기** | `KEYS:` / `키:`, `DEATH:` / `사망:`, `best:` / `최고:`, `←/→ move` / `←/→ 이동`, `pushed off the top` / `위로 밀림`, `fell through all platforms` / `모든 발판 통과 추락` |
| 액션 라벨·시작 안내 | **ko/en 분기** | `[ > retry ]` / `[ > 다시 ]`, `[ > home ]` / `[ > 홈 ]`, `press [Enter] to start_` / `[Enter] 키를 눌러 시작_`, `tap to start` / `탭하여 시작` |

#### 분기 규칙 한 줄 요약

- **터미널이 출력하는 것** → 영어 고정 (실제 터미널처럼 보여야).
- **사람이 사람에게 안내하는 것** → ko/en 분기 (한국어 사용자 배려).

#### Initial 화면 한국어 버전 (예시)

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  $ fall -f                                               │
│                                                          │
│  ── fall -f v1.0.0 — follow the fall ─────────────────   │
│                                                          │
│  키:    ←/→   이동                                        │
│  사망:  [TIMEOUT]   위로 밀려 사라짐                       │
│         [SEGFAULT]  발판 모두 놓치고 추락                  │
│                                                          │
│  최고:  256 라인                                          │
│                                                          │
│  $ [Enter] 키를 눌러 시작_                                │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

명령어 prompt(`$ fall -f`), 헤더, 부제목, 시스템 토큰(`[TIMEOUT]`/
`[SEGFAULT]`)은 영어 고정. 그 외 라벨·설명·시작 안내는 한국어로 표시.

> 본 정책은 본 게임 한정. 서비스 공통 영역 (헤더, 푸터, 카탈로그 페이지의
> 사이트 공통 메뉴 등) 은 기존 i18n 정책 (CLAUDE.md) 그대로 따른다.

### 2.17 라인 끝 마커 정책

- **라인 끝에 별도 마커(▌ 등)를 붙이지 않는다.** 텍스트의 마지막 글자가
  그대로 발판의 우측 끝.
- 이유: "그냥 빌드 로그" 위장 효과 최우선. 실제 터미널 출력에 없는
  글리프를 추가하면 위장이 깨진다.
- 본 문서의 §1 / §1.2 / §1.3a / §3.x 의 ASCII 스케치에서 `▌` 가 등장
  하는 부분은 **시각 보조용 표시일 뿐** 이며 실제 게임에서는 렌더링되지
  않는다.

### 2.6 출시 범위

- **첫 게임은 Falldown 류 1개**. 슬러그·이름은 §2.8 참조 (`/game/fall-f`).
- 출시 후 `/game` 카탈로그에서 추가 게임을 점진 확장 (P3 이상).
- 카탈로그 페이지는 게임이 1개일 때도 그 형태를 유지 — 미래 확장을
  전제한 페이지 구조를 처음부터 갖춘다.

### 2.7 메커니즘 세부 — 카드 #1 Pure Falldown 베이스

- 입력은 ←/→ 만. 점프·호핑 없음. 자유낙하 + 라인 위에 닿으면 라인이
  위로 흐르는 동안 같이 끌려 올라간다.
- 사망: `[TIMEOUT]` (위로 밀려 사라짐) / `[SEGFAULT]` (바닥 추락).
- 위장 효과 최우선. "그냥 터미널이 흐른다" 인상.

#### 독창성 — Falldown 장르 자체는 기존 (Fall Down, NS-Shaft 등)이므로,
독창성은 **"발판이 실제 텍스트"** 라는 재료에서 나온다. 추가 옵션은
미래 확장 후보로 §2.7a 에 보존.

#### §2.7a 미래 확장 옵션 (첫 출시에는 포함하지 않음)

- (a) **타이핑 등장**: 라인이 한 번에 등장하지 않고 타이핑 효과로 점점
  길어진다. 발판 폭이 시간 경과에 따라 늘어남. 터미널 미학과 게임
  긴장감을 동시에 잡는다.
- (b) **피드 라인 차용**: API down 일 때 피드의 실제 포스트 제목/본문
  일부를 발판 텍스트로 사용. "같은 처지의 사람들 외침을 밟고 떨어진다."
  단 본문 길이 정규화·필터링 필요.
- (c) **토큰 보너스**: 라인 안 특정 토큰(`$`, `>`, `npm`, `error`)
  글리프 정확히 위에서 통과 시 +1 보너스. 가독성을 해치지 않도록
  보너스 시각 표시는 매우 절제 (잠시 글리프 색만 변경).

#### 난이도 곡선

- "자연스럽고 너무 어렵지 않게." 기본 곡선:
  - 0–20초: 1.0 line/sec, 라인 길이 균등 (16~40 chars).
  - 20–60초: 1.3 line/sec, 라인 길이 분포 약간 넓힘 (10~50).
  - 60초+: 1.6 line/sec 캡, 라인 길이 (8~60).
- "갑자기 어려워지는" 구간 없음. 모든 변화는 선형 또는 부드러운 ease-in.

### 2.8 첫 게임 슬러그·표시 이름

- 슬러그: **`/game/fall-f`**.
- 표시 이름: **`fall -f`** (Unix 명령어 풍, 띄어쓰기 포함).
- 부제목: **"follow the fall"** — `tail -f` 의 `-f`(`--follow`) 패러디.
  "떨어짐을 끝까지 따라간다" 의 의미.
- **부제목은 영어 고정** (ko 번역 안 함). 게임 컨셉(터미널 풍) 과 영어
  명령어·옵션 구조의 조어가 잘 맞고, 마땅한 한국어 번역은 모두 매력이
  떨어진다. UI 카피 분기 정책은 §2.19 참조.

### 2.9 게임 시작 트리거 (즉시 시작)

- 진입 시 Initial 화면(§2.18) 노출. 마지막 줄에
  `$ press [Enter] to start_` 프롬프트.
- **Enter 입력 → 즉시 게임 시작.** 카운트다운(3-2-1-GO) 은 채택하지
  않는다. 큰 글자 사이즈 변화가 모노스페이스 / 터미널 미학을 깨고,
  명령어 실행 직후 결과가 즉시 흐르는 터미널의 자연스러움과도 어긋난다.
- Enter 시점에 첫 라인이 화면 아래에서 등장하기 시작하고 캐릭터가
  화면 상단에서 자유낙하를 시작. 둘 다 동시.
- 재시작 동선도 동일: `[ > retry ]` 클릭 또는 Enter → 즉시 시작.
- 모바일 동등성: `press [Enter]` → `tap to start`. 탭 → 즉시 시작.

### 2.10 게임 종료 액션 (점수 공유 제거)

- 게임 종료 화면에 노출되는 액션:
  - `[ > retry ]` — 즉시 재시작 (디폴트 포커스).
  - `[ > home ]` — `/` 로.
- 점수를 피드에 공유하는 기능(`[ > brag in feed ]`)은 **제거**. 이유:
  피드 자체가 API 상태에 종속(SERVICE_PLAN §4.1)이라 게임이 API 상태에
  관여하게 되기 때문. §2.3 의 분리 원칙에 따라 게임은 게임에만 집중.
- 따라서 닉네임 변경 등 sessionStore 변경 흐름도 본 게임 범위에서는
  발생하지 않는다. 기존 [`SERVICE_PLAN.md`](./SERVICE_PLAN.md) §4.3 의
  닉네임 정책은 변경 없이 유지.

### 2.11 모바일 컨트롤 (잠정)

- `fall -f` 의 ←/→ 입력은 모바일에서 화면 좌/우 절반 영역 탭으로 매핑.
  한쪽을 길게 누르면 그 방향으로 계속 이동. 두 영역 동시 터치는 무시.
- 진입 직후 Initial 화면에서는 "tap to start" 만 활성. 게임 시작 후
  좌/우 탭 영역 활성.

### 2.12 카탈로그 페이지 디자인

`/game` 진입 시 `ls -la /game/` 출력 풍 테이블.

```
$ ls -la /game/

-rwxr-xr-x  fall-f             2026-05  follow the fall
-rwxr-xr-x  (more coming...)

> _
```

- 첫 컬럼: 권한 풍 데코레이션 (`-rwxr-xr-x` — 실행 파일 풍).
- 둘째 컬럼: 게임 파일명. 클릭 시 `/game/{slug}`. 슬러그가 `fall-f` 라
  파일명도 `fall-f`. 확장자(`.exe`) 는 붙이지 않음 — Unix 풍에 가깝게.
- 셋째 컬럼: 출시 연-월.
- 넷째 컬럼: 부제목 (영어 고정, §2.8).
- 마지막 줄에 깜빡이는 프롬프트(`> _`) — 페이지 전체에 살아 있는 인상.
- 게임이 1개일 때도 같은 형식 유지. 두 번째 줄 `(more coming...)` 으로
  확장 가능성을 시사.

### 2.13 사망 결과 화면

`core dumped` 계열 가짜 스택트레이스 풍.

```
[SEGFAULT] segmentation fault (core dumped)

  at line 142, score 99
  best  256
  in    fall_f::on_tick (game.ts:42)

[ > retry ]  [ > home ]
```

- `[TIMEOUT]` 사망 시: 타이틀만 `[TIMEOUT] line scroll past viewport`
  로 교체.
- 가짜 스택트레이스는 진짜 코드 라인을 노출하지 않음. 일관된 가짜
  포맷 사용 (`fall_f::on_tick (game.ts:42)`). 함수명에 하이픈을 못 쓰는
  실제 코드 관행에 맞춰 `fall_f` 표기.
- 디폴트 포커스는 `[ > retry ]` — Enter 한 방으로 재시작.

### 2.14 API 복구 토스트 — **제거**

이전 결정으로 채택했던 "down → normal 전환 시 토스트" 는 §2.3 의
"게임은 API 상태와 분리" 원칙에 따라 **제거** 한다. 게임 내부에서 API
상태에 어떠한 반응도 하지 않는다.

### 2.15 독창성 옵션 정책

- §2.7a 의 (a) 타이핑 등장 / (b) 피드 라인 차용 / (c) 토큰 보너스 는
  **첫 출시에 포함하지 않는다.**
- §2.7a 자체는 미래 확장 후보로 본 문서에 그대로 보존.
- 출시 후 사용자 반응을 본 뒤 하나씩 점진 추가.

### 2.16 본 결정의 잠정성

- 사용자 의사: "무슨 게임을 만들고 싶은지 아직 명확하게 보이지 않는다."
  **§2 의 결정들은 이를 전제한 첫 출발점이며**, 실제 ASCII 시각 시뮬
  레이션 또는 작은 프로토타입을 본 뒤 §2.7 메커니즘 / §2.13 결과 화면
  등 일부 항목은 갱신될 수 있음.
- 따라서 본 문서는 "아이디어 합의 1차본" 으로 간주하고, 다음 단계는
  **실제 움직이는 모습** 을 보고 재검토하는 것.

## 3. 검토 자료 (Reference / not yet decided again)

### 3.1 게임이 노출되는 시점·위치 — **결정됨 (§2.1, §2.2, §2.3 참조)**

`/game` 카탈로그 + `/game/:slug` 인스턴스, 헤더에 `[ play ]` 진입점,
normal/down 양쪽에서 접근 가능. 자세한 내용은 §2 참고.

### 3.2 메커니즘 방향성 (자세히)

원안: "수직 스크롤 + 텍스트 라인의 끝이 플랫폼". 이걸 게임으로 풀려면
"라인이 어느 방향으로 흐르고, 플레이어는 무엇을 하는가" 두 가지가
정해져야 한다. 두 갈래의 후보를 풀어 설명한다.

#### 옵션 A — Falldown 형 (라인은 위로 흐르고, 플레이어는 그 위에서 버틴다)

비유: 옛날 플래시 게임 "Falldown" / 모바일 "NS-Shaft" 의 텍스트 버전.

```
[ T = 0초 ]                       [ T = 1초 ]                       [ T = 3초 ]
┌─────────────────────────┐       ┌─────────────────────────┐       ┌─────────────────────────┐
│                         │       │ $ npm run build      ▌  │       │ $ tsc --noEmit       ▌  │
│                         │       │                         │       │ $ npm run build  ▌@     │ ← 이 라인은 곧 화면 위로
│                         │       │                         │       │ ✓ 142 modules    ▌      │
│ $ npm run build      ▌  │       │  @                      │       │ ▸ chunks...      ▌      │
│                         │  →    │                         │  →    │                         │
│                         │       │                         │       │  @ ← 다음 라인 기다림
└─────────────────────────┘       └─────────────────────────┘       └─────────────────────────┘
새 라인이 아래에서 등장             1초 뒤 위로 한 칸 스크롤             3초 뒤 여러 라인 누적, 가장 위 라인은 곧 사라짐
```

- 카메라는 고정. 새 라인이 화면 아래쪽에서 한 줄씩 등장해 위로
  스크롤된다 (실제 터미널 출력과 동일한 흐름).
- 플레이어 `@` 는 자유낙하. 좌우(←/→)로만 위치를 조정한다.
- 떨어지다가 텍스트 라인 위(라인 시작 X ~ 라인 끝 X 범위)에 닿으면 그
  위에 착지. 라인이 위로 스크롤되면 플레이어도 같이 위로 끌려 올라간다.
- 라인이 위로 너무 많이 밀려서 플레이어가 화면 위쪽으로 사라지면 →
  `[TIMEOUT]`.
- 떨어질 때 어떤 라인 위에도 착지하지 못하고 화면 바닥까지 가면 →
  `[SEGFAULT]`.
- 핵심 입력은 **좌우 이동만**. 점프 키는 없거나, 옵션으로 작은 호핑 추가.
- **장점**: "터미널에 로그가 한 줄씩 찍힌다" 인상과 100% 일치 — 멀리서
  보면 빌드 로그를 실시간으로 보고 있는 사람으로 보임.
- **단점**: 점프 같은 다이내믹한 입력이 없어 액션감이 단조로울 수 있다.
  대신 라인 길이/등장 속도를 변주해 난이도를 만든다.

#### 옵션 B — Doodle Jump 형 (플레이어가 위로 점프, 카메라가 따라간다)

비유: 모바일 게임 "Doodle Jump" 의 텍스트 버전.

```
[ T = 0초 ]                       [ T = 2초 ]                       [ T = 4초 ]
┌─────────────────────────┐       ┌─────────────────────────┐       ┌─────────────────────────┐
│                         │       │ $ make all       ▌      │       │ $ ./build.sh         ▌  │
│ $ make all       ▌      │       │                         │       │                         │
│   @ ← 점프!             │       │ $ tsc -p .       ▌      │       │ $ make all       ▌      │
│                         │       │      @ ← 점프 도중      │       │ $ tsc -p .       ▌@     │
│ $ tsc -p .       ▌      │  →    │                         │  →    │ $ npm test       ▌      │
│                         │       │ $ npm test       ▌      │       │                         │
│ $ npm test       ▌      │       │                         │       │ ↓ 아래 라인은 사라져감
└─────────────────────────┘       └─────────────────────────┘       └─────────────────────────┘
플레이어가 점프해 올라감           카메라가 따라 위로 이동              아래쪽 옛 라인은 화면 밖으로 사라짐
```

- 라인이 화면 위쪽에 펼쳐져 있고 플레이어는 ←/→ + Space 점프(또는 자동
  점프) 로 위로 올라간다.
- 카메라가 플레이어를 따라가면서 화면 위쪽에 새 라인이 계속 생성됨,
  아래쪽 라인은 사라짐 (= 시각적으로 화면이 위로 흘러감).
- 점프 도중 어떤 라인에도 착지하지 못하고 화면 바닥으로 떨어지면 →
  `[SEGFAULT]`.
- **장점**: 점프 액션이 명확해 게임적 즐거움이 크다.
- **단점**: 화면이 "위로 올라가는" 인상은 일반 터미널의 출력 흐름과
  반대 방향이라, "비개발자가 보면 빌드 로그처럼 보인다" 효과가 약해진다.

#### 정리

| 항목 | A (Falldown) | B (Doodle Jump) |
|---|---|---|
| 터미널 미학 정합성 | ★★★★★ | ★★ |
| 액션감 | ★★ | ★★★★ |
| 입력 단순성 | ★★★★ (←/→ 만) | ★★★ (←/→ + 점프) |
| 위장 효과 (멀리서 코딩처럼 보임) | ★★★★★ | ★★ |

> 채택: **A (Falldown).** 본 서비스의 핵심 가치 명제와 정합. (§2.7)

### 3.2b Falldown 류 세부 변주

§2.6 에서 첫 게임을 Falldown 류로 확정했고, 그 안의 세부 룰 카드 3종을
비교 후 §2.7 에 카드 #1 Pure 를 베이스로 채택.

#### 카드 #1 — Pure Falldown (가장 미니멀, 채택)

```
규칙
  - 점프 없음. 플레이어는 항상 자유낙하.
  - 좌우 이동(←/→)만 허용.
  - 라인 위에 닿으면 그 라인이 위로 흐르는 동안 같이 위로 끌려감.
  - 라인 등장 속도: 첫 30초 1.0 line/sec → 30초 후 1.5 → 60초 후 2.0.
  - 모든 라인 길이는 동일 분포 (8~40 chars 범위 균등).
  - 사망: 화면 위로 1행 이상 밀리면 [TIMEOUT]. 바닥 닿으면 [SEGFAULT].
인상
  - 가장 단순. "터미널이 흘러가는데 가만히 있으면 죽는" 게임.
  - 위장 효과 최강. 학습 곡선 5초.
```

> 실제 §2.7 의 난이도 곡선은 이 카드의 기본값보다 더 완만하게 조정됨
> (사용자 피드백 "너무 어렵지 않게" 반영).

#### 카드 #2 — Hop Falldown (작은 자동 호핑, 미래 확장)

```
규칙
  - 라인 위에 착지하면 자동으로 작은 호핑(0.3초 주기 위로 살짝).
  - 호핑 덕에 다음 라인이 약간 위에 있어도 잡을 수 있음.
  - 좌우 이동(←/→)만 입력. 점프 키 없음.
  - 라인 길이는 짧음(6~14)/중간(15~30)/김(31~60) 3-tier 혼합.
  - 사망: 카드 #1 과 동일.
인상
  - 카드 #1 보다 살짝 더 다이내믹. "텍스트 위에서 살짝 발 구르며
    다음 라인을 기다리는" 그림.
```

#### 카드 #3 — Manual Jump Falldown (수동 점프 + 변종 라인, 미래 확장)

```
규칙
  - 좌우 이동(←/→) + Space 점프(수직 점프 1회, 라인 위에서만 발동).
  - 라인 종류:
    - 일반 라인 — 평범한 발판.
    - [OK] 라인 — 발판 + 점수 +5 (한 번 닿으면 다시 못 받음).
    - [ERR] 라인 — 한 번 닿으면 부서지는 발판 (1초 후 사라짐).
    - [WARN] 라인 — 발판 폭이 좁음(4 chars 이하).
  - 라인 등장 속도는 카드 #1 과 동일.
  - 사망: 카드 #1 과 동일.
인상
  - 가장 풍성한 게임플레이. 단 입력이 점프까지 늘어나 학습 곡선↑.
  - "비개발자가 보면 코딩 중" 위장 효과는 카드 #1 보다 살짝 약함
    ([OK]/[ERR] 색이 두드러지면 게임처럼 보일 수 있음).
```

#### 비교 표

| 항목 | #1 Pure | #2 Hop | #3 Manual |
|---|---|---|---|
| 입력 키 | ←/→ | ←/→ | ←/→ + Space |
| 학습 곡선 | ★ (5초) | ★★ (15초) | ★★★ (1분) |
| 위장 효과 | ★★★★★ | ★★★★ | ★★★ |
| 게임적 재미 | ★★ | ★★★ | ★★★★ |
| 구현 복잡도 | 낮음 | 중간 | 중간-높음 |
| 첫 출시 적합도 | ★★★★★ | ★★★★ | ★★★ |

### 3.3 플랫폼의 정확한 정의 — **결정됨: 3.3a 라인 전체가 발판**

원안: "텍스트 오른쪽 **끝나는 지점**이 플랫폼이 되는".

§2.17 의 결정으로 **라인 끝 마커(▌)를 표시하지 않으므로**, 끝 부분만
발판으로 잡는 3.3b 변종은 시각적으로 표시할 수단이 없다. 따라서
**3.3a (라인 전체가 발판)** 로 자연 귀결.

- 발판 X 범위 = 라인의 시작 X (들여쓰기 포함) ~ 마지막 글자 X.
- 라인 길이가 곧 발판 폭. §1.3a 의 들여쓰기 변주를 함께 적용해
  발판 시작 X 도 다양화 → 좌우 양방향 게임플레이 가능.
- 변동성은 라인 길이 + 들여쓰기 두 변수에서 동시에 나오므로 발판이
  단조롭지 않다.

#### 3.3a — 라인 전체가 플랫폼 (이전 잠정 채택)

- 텍스트가 그려진 X 범위(0 ~ line_length) 가 그대로 발판.
- 라인 길이가 곧 발판 폭. 짧을수록 어렵다.
- 게임적으로는 발판이 항상 왼쪽 정렬이므로, 좌우 이동의 의미가
  "발판 위에 머물기" 보다 "발판이 닿는 곳까지 가기" 에 가까움.

```
$ npm run build ▌                            │
████████████████          ← 이 X 구간 전체 발판
```

#### 3.3b — 라인 끝 근처 / 끝 한 점만 발판

- 라인의 끝 부분(예: 마지막 N글자, 또는 단 한 칸)만 실제 발판.
- 텍스트 본문 위는 떨어지면 통과해서 추락.
- "끝 지점이 발판" 이라는 원안 어감과 가장 일치.
- 게임적으로 좌우 이동의 의미가 살아남: "각 라인의 끝 X 좌표를 잘
  맞춰서 떨어진다" 가 핵심 액션.
- 단점: 발판이 좁아 난이도가 올라감. 너무 어렵지 않게 하려면 끝
  근처 N 칸을 발판으로 잡거나, 발판 폭을 시각적으로 강조하는 글리프
  변형 필요.

```
$ npm run build ▌                            │
                ██            ← 끝 한 칸~몇 칸만 발판
```

##### 3.3b 의 변종

- (3.3b-i) **끝 한 칸 (▌ 자체만)** — 가장 핵심 충실. 매우 어려움.
- (3.3b-ii) **끝 5칸** — 사용자 원안 + 적당한 난이도. 현실적인 후보.
- (3.3b-iii) **끝 N칸 (라인 길이의 30% 등)** — 라인 길이가 길수록
  발판도 길어짐. 짧은 라인은 어려운 발판, 긴 라인은 쉬운 발판.

#### 결정 보류

- 본 문서는 일단 **3.3b-ii (끝 5칸)** 를 유력 후보로 두되, 다음 단계
  시각 시뮬레이션에서 두 해석을 모두 그려보고 비교 후 확정.
- 시각 스케치(§1) 는 두 해석 중 어느 쪽으로 가도 동일하게 "라인이 짧으면
  발판 위치도 왼쪽" 이라는 큰 규칙은 유지된다.

### 3.4 골(goal)·엔딩 (자세히)

#### G-1. 무한 생존형 (= 점수 도전)

- 죽을 때까지 끝없이 진행. 점수는 "살아남은 라인 수" 또는 누적 시간.
- 사망 시 화면: `[CRASHED] survived: 142 lines / best: 188`.
- 다시 시작은 `> retry_` 프롬프트.
- 비유: Tetris 의 점수 도전 모드, Doodle Jump 의 기본 모드.

#### G-2. 스테이지 클리어형

- "build 100 modules" 같은 명확한 도달 목표. N 라인 통과 시 클리어 화면.

#### G-3. API 복구 자동 종료 타임어택 (down 상태 한정)

- API 가 down → normal 로 회복되는 순간 게임이 자동으로 끝나고 점수
  표시.

#### G-4. 하이브리드 (G-1 + G-3 의 토스트 안내) — **채택 (§2.4)**

- 기본은 G-1 (무한 생존). 죽을 때까지 점수 누적.
- API 가 down → normal 로 바뀌면 게임 화면 위로 토스트.
- "go code" 클릭 → `/` 로 이동. "keep playing" 클릭 → 토스트 닫고
  그대로 계속.

### 3.4a 플레이어 캐릭터 (자세히)

| 후보 | 모양 | 인상 |
|---|---|---|
| **`@`** (채택) | `@` | Roguelike (Nethack, Rogue, Angband) 의 전통 플레이어 표현. 한 글자라 텍스트 흐름 속에서 시각적으로 또렷이 분리. |
| **`▌` 또는 `█`** | `█` | 터미널 커서 그 자체. 라인 끝 플랫폼 표시(▌)와 글리프가 충돌할 수 있음. |
| **`>`** | `>` | icantcode 의 입력 prompt 기호 재활용. 다만 화살표처럼 보여 캐릭터 인식이 약함. |
| **`◆` / `▲`** | `◆` | 별도 도형. 모노스페이스 미학에선 살짝 튄다. |

### 3.5 죽음·실패 조건

- 플랫폼을 놓치고 화면 바닥으로 떨어짐 → `[SEGFAULT]`.
- 라인이 위로 밀려 플레이어가 화면 밖으로 사라짐 → `[TIMEOUT]`.
- 사망 시 `core dumped` 풍 메시지 (§2.13). 다시 시작은 `> retry_`.

### 3.6 컨트롤

- 데스크탑: ←/→ 이동.
- 모바일: 화면 좌/우 탭 영역 (§2.11).
- 키 가이드는 시작 화면에 `[KEYS] ←/→ move` 식으로 노출.

### 3.7 라인 텍스트 컨텐츠

- npm/vite/tsc/git 같은 **실제 빌드/lint/테스트 로그 풍** 가짜 텍스트.
- 이따금 icantcode 컨텍스트 라인 (`[OK] feed warm`, `> @anon posted`)

- 다국어: ko/en 분기 (i18n 키로 관리). 단, 텍스트가 너무 짧으면 게임성에

### 3.8 시각·청각 모션 정책

- 라인 등장: 타이핑 효과 없이 한 번에 등장 (게임 페이스 우선). 단,
  사망 직후 다음 게임 시작 시 첫 라인은 타이핑 효과로 들어와도 OK.
- 캐릭터(`@`): 가만히 있을 때 커서 깜빡임 적용.
- `prefers-reduced-motion` 시: 라인 스크롤이 step 단위로 점프 (애니메이션
  대신 즉시 이동). 게임 자체를 disable 하지는 않음 — 게임이 핵심
  컨텐츠가 아니라 보조이므로.
- 사운드 없음. 터미널 미학과 어긋나고, 불의의 다운타임에 다른 사람이
  옆에 있을 가능성 고려.

  게임 엔진 없이 React + `requestAnimationFrame` + Tailwind 로 충분히

2. **출시 후 독창성 옵션 우선순위** — (a) / (b) / (c) 중 어느 것이
   먼저 들어갈지는 출시 후 결정.
3. **점수 데이터 영속성** — 세션 메모리 한정 유지로 충분한지, 아니면
   localStorage 에 best score 만이라도 둘지 (현재 정책: 세션 한정).
4. **i18n 카피** — `[SEGFAULT] segmentation fault (core dumped)` 같은
   문구는 영어 그대로 둘지, 한국어로 일부 번역할지.
